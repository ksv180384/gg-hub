# Вращение рулетки (spin wheel)

Документ описывает, как на фронтенде и на сокет-сервере задаётся и анимируется поворот колеса, как вычисляется победитель и как клиенты остаются синхронными.

## Участники

| Компонент | Роль |
|-----------|------|
| `frontend/src/widgets/spin-wheel/model/useSpinWheel.ts` | Математика угла, RAF-анимация, локальный и серверный спин |
| `frontend/src/widgets/spin-wheel/ui/SpinWheel.vue` | Canvas: `rotate(angle)`, отрисовка сегментов, стрелка сверху |
| `frontend/src/shared/lib/useGuildAuctionWheelSocket.ts` | Socket.IO: комната гильдии, `auction:spin`, `spinFromServer` |
| `socket_server/src/auctionSocketHandler.js` | Один общий payload вращения для всех в комнате |

## Геометрия и система углов

- Сегменты нумеруются **0 … n−1**. Каждому соответствует дуга **`arc = 360 / n`** градусов.
- **Граница остановки** в «пространстве колеса» задаётся углом **`norm`** (градусы на окружности сегментов): попадание в интервал сегмента `winIdx` с отступами от краёв (`margin`, `span`), чтобы стрелка не липла к границе.
- **Стрелка** в UI направлена **вверх** (на 12 часов). В коде отрисовки верх соответствует **270°** в системе угла колеса после поворота canvas.
- **`angle`** (ref в `useSpinWheel`) — суммарный поворот canvas в **градусах** (может быть ≫ 360 из‑за полных оборотов).

Связь «под стрелкой нужен сегмент с центром в `norm`»:

- Целевой **остаток** поворота по модулю 360:  
  `remainderTarget = mod360(270 − norm)`  
  (так при данном `angle mod 360` под стрелкой оказывается нужная отметка на колесе.)

## Вычисление финального угла

На каждый спин задаются:

1. **`startAngle`** — текущее значение `angle` в начале анимации.
2. **`remainderTarget`** — какой остаток `angle mod 360` нужен в конце (из `norm` и геометрии стрелки).
3. **`delta = mod360(remainderTarget − mod360(startAngle))`** — минимальный неотрицательный поворот по модулю 360 до совпадения остатка (без лишнего «назад»).
4. **`fullTurns`** — целое число **полных** оборотов **360°** «сверху», для зрелищности и чтобы итог не выглядел зацикленным на одном обороте.

**Финальный угол:**

```text
finalAngle = startAngle + delta + 360 * fullTurns
```

Пока `ease` в анимации идёт от 0 до 1, текущий угол:

```text
angle = startAngle + ease * (finalAngle − startAngle)
```

В конце кадра с `progress === 1` выставляется ровно `finalAngle`.

### Локальный спин (`spin`)

- Случайный `winIdx`, случайный `norm` в допустимом диапазоне сегмента (как на сервере).
- **`fullTurns`** = `fullTurnsForDurationMs(duration)` — зависит от выбранной длительности (см. ниже).

### Синхронный спин (`spinFromServer`)

- **`norm`**, **`fullTurns`**, **`duration`** приходят с сервера (и независимое поле `winIdx` в payload для отладки/логов; угол считается по **`norm`**).
- `norm` ограничивается допустимым диапазоном на клиенте на случай рассинхрона версий.

## Полные обороты и длительность

При большой длительности (например 60 с) тот же нормированный прогресс `ease ∈ [0,1]` растягивает **один и тот же** диапазон угла `Δ = finalAngle − startAngle` на большее время → средняя угловая скорость падает. Чтобы длинный спин не выглядел «ползущим», число оборотов **увеличивается** с ростом `duration`:

- **Сокет:** `randomFullTurnsForDuration(durationMs)` в `auctionSocketHandler.js`.
- **Фронт (локальный спин):** `fullTurnsForDurationMs` в `useSpinWheel.ts`.

Формулы **должны совпадать**: **`N = round(8 × T / 4 с)`**, затем **`clamp(N, 5, 120)`**. Тогда полный угол **`∝ T`**, и на линейном круизе **`ω ≈ 360×N/T ≈ const`** — длительность 60 с не выглядит медленнее 4 с по скорости вращения (раньше `N` рос слабее `T`, из‑за этого 60 с «ползло»).

## Кривая анимации (ease по времени)

Нормированное время **`t = elapsed / T`**, где `T` — полная длительность анимации в мс.

### Три фазы по календарю

| Фаза | Доля времени | Доля полного угла | Формула |
|------|----------------|-------------------|---------|
| Разгон | **0–20%** | **10%** | `ease = 0.1 × (t/0.2)³` — **ω=0** в `t=0`, плавный рост без рывка; к `t=20%` производная совпадает с круизом. |
| Круиз (макс. ω) | **20–70%** | **75%** | Линейно: наклон **1.5** (доля угла / доля времени) — **самый длинный** участок при максимальной скорости. |
| Торможение | **70–100%** | **15%** | **`spinTailDecelBlend`**: **`1 − (1−v)³`** (кубический ease-out), стык C¹ с круизом; монотонно, без вылета **φ>1** и обратного вращения. |

Первые **10%** по времени входят в кубический разгон (совместно с **10–20%**), чтобы к началу круиза уже выйти на ровную максимальную скорость без отдельного «второго разгона» на 3-й секунде.

### Итог

```text
ease(t) ∈ [0, 1]  →  angle = startAngle + ease * (finalAngle − startAngle)
```

При **`t = 1`** всегда **`ease = 1`**.

## Цикл анимации

- Используется **`requestAnimationFrame`**: на каждом кадре считается `elapsed`, `ease`, обновляется **`angle`**, перерисовывается canvas (`watch(angle)` в `SpinWheel.vue`).
- **Обратный отсчёт:** `spinCountdownSeconds ≈ ceil((T − elapsed) / 1000)` до завершения.
- По завершении вызывается **`calculateResult`**: по текущему `angle` и тому же `topAngle = 270` вычисляется индекс сегмента под стрелкой и пишется **`result`**.

## Синхронизация по сокету

1. Клиент с правом управления шлёт **`auction:spin-request`** с **`durationMs`** (кламп на сервере 2000–60000 мс).
2. Сервер один раз вызывает **`buildSpinPayload(n, durationMs)`** и шлёт всем в комнате **`auction:spin`** с полями **`norm`**, **`fullTurns`**, **`duration`**, плюс **`winIdx`**.
3. Все клиенты вызывают **`spinFromServer(payload)`** с одинаковыми числами → одинаковая траектория и итог.

При изменении логики оборотов или геометрии нужно обновлять **и** фронт, **и** `auctionSocketHandler.js`, и этот документ.
